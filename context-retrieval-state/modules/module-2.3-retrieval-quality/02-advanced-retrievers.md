## Real-World Analogy

A naive library search: look for books with titles matching your exact words.
An expert librarian search: look for synonyms, check summaries, cross-reference related topics,
then return only the most relevant sections — not the entire book.

The three advanced retrievers map to three different expert-librarian strategies.

---

## Overview of the Three Advanced Retrievers

```
┌────────────────────────────────────────────────────────────────────────┐
│ Retriever                  │ Problem it solves        │ Mechanism      │
├────────────────────────────┼──────────────────────────┼────────────────┤
│ MultiQueryRetriever        │ Vocabulary mismatch      │ Rephrase query │
│                            │ (user uses different     │ multiple ways; │
│                            │  words than document)    │ union results  │
├────────────────────────────┼──────────────────────────┼────────────────┤
│ ContextualCompression      │ Retrieved chunk contains │ LLM compresses │
│ Retriever                  │ relevant + irrelevant    │ chunk to only  │
│                            │ content mixed together   │ relevant part  │
├────────────────────────────┼──────────────────────────┼────────────────┤
│ ParentDocumentRetriever    │ Small chunk lacks context│ Index small;   │
│                            │ (chunk has answer but    │ return large   │
│                            │  not the surrounding     │ parent section │
│                            │  context)                │                │
└────────────────────────────┴──────────────────────────┴────────────────┘
```

## MultiQueryRetriever — Expand the Query

### The Problem

Embedding similarity is sensitive to vocabulary.
"Password reset" and "credential recovery" may not be similar in embedding space
even though they mean the same thing.

### The Solution

Generate multiple rephrasings of the question, run all of them against the vector store,
and take the union of results.

```
Original: "How do I reset my password?"

Rephrasings generated by LLM:
  1. "What is the account credential recovery procedure?"
  2. "How can I regain access to my account?"
  3. "Steps to change forgotten account password"
  4. "Account unlock and password recovery guide"

Each rephrase is embedded and searched separately.
The union of all results gives much broader coverage.
```

### Code

```python
from langchain.retrievers import MultiQueryRetriever
from langchain_openai import ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings
import logging

# Enable logging to see the generated queries (useful during development)
logging.basicConfig()
logging.getLogger("langchain.retrievers.multi_query").setLevel(logging.INFO)

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = FAISS.load_local(
    "./faiss_index",
    embeddings,
    allow_dangerous_deserialization=True,
)

# The query LLM generates rephrasings — use a cheap, fast model
query_llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

multi_retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(search_kwargs={"k": 3}),
    # ↑ base retriever: used for each generated query variant
    # k=3 per query × 3-5 variants = 9-15 candidate chunks before deduplication

    llm=query_llm,
    # include_original=True: also run the original query (recommended)
    # Default: True
)

# Invoke — returns deduplicated union of all queries' results
docs = multi_retriever.invoke("How do I reset my password?")
print(f"Retrieved {len(docs)} unique chunks")
# Typically 5-10 diverse, relevant chunks vs 3-4 with simple similarity search

# When MultiQueryRetriever runs, you'll see log output like:
# INFO:langchain.retrievers.multi_query: Generated queries:
# ['What is the password recovery process?',
#  'Steps to regain access to my account',
#  'Account password reset instructions']
```

### When to Use MultiQueryRetriever

- When users ask questions using different vocabulary than your documents use.
- When retrieval is missing relevant chunks that contain the right content but wrong words.
- Cost trade-off: each query = 1 extra LLM call + k extra embedding searches.

## ContextualCompressionRetriever — Filter to the Relevant Part

### The Problem

A retrieved chunk may be 500 characters of mixed content.
Only 50 characters are actually relevant to the question.
The other 450 characters add noise, dilute the model's focus, and waste tokens.

### The Solution

After initial retrieval, pass each chunk through a compressor LLM.
The compressor extracts only the sentences relevant to the query.

```
Before compression:
  Chunk: "Employees are entitled to three weeks of vacation after the first year.
           Performance reviews are held annually in December.
           The overtime rate is 1.5x for hours exceeding 40 per week.
           Company cars must be returned within 30 days of termination."

Query: "What is the overtime rate?"

After compression:
  "The overtime rate is 1.5x for hours exceeding 40 per week."
  ↑ Only the relevant sentence — the rest is stripped out.
```

### Code

```python
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor
from langchain_openai import ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = FAISS.load_local(
    "./faiss_index",
    embeddings,
    allow_dangerous_deserialization=True,
)

# The compressor LLM — used to extract relevant content from each chunk
compressor_llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

# LLMChainExtractor uses the LLM to extract only relevant sentences from each chunk
compressor = LLMChainExtractor.from_llm(compressor_llm)

compression_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=vectorstore.as_retriever(search_kwargs={"k": 6}),
    # ↑ Retrieve 6 chunks initially (more than you'll keep)
    # After compression, you'll have 3-5 focused excerpts
)

# Invoke
compressed_docs = compression_retriever.invoke("What is the overtime rate?")
for doc in compressed_docs:
    print(f"Source: {doc.metadata.get('source')}, Page: {doc.metadata.get('page')}")
    print(f"Compressed content: {doc.page_content}")
    # "The overtime rate is 1.5x for hours exceeding 40 per week."
    print("---")
```

### When to Use ContextualCompressionRetriever

- When retrieved chunks are large and contain mixed topics.
- When token budget is tight (compressed chunks use fewer tokens).
- Cost trade-off: one LLM call per retrieved chunk (expensive at high k).

## ParentDocumentRetriever — Index Small, Return Large

### The Problem

Small chunks index well (precise retrieval) but lack context.
Large chunks have context but index poorly (less precise retrieval).

### The Solution

Use the best of both:

- **Index** small child chunks for precise retrieval.
- **Return** the large parent chunk that contains the child.

```
Document structure:
  Parent section (1000 chars):
    "Section 3.4: Overtime Policy
     3.4.1 Eligibility: Non-exempt hourly employees are eligible for overtime.
     3.4.2 Rate: Overtime is compensated at 1.5x the regular hourly rate.
     3.4.3 Threshold: Overtime applies to hours worked beyond 40 in a work week.
     3.4.4 Approval: Overtime must be pre-approved by a direct manager."

  Child chunks (200 chars each):
    Child A: "3.4.1 Eligibility: Non-exempt hourly employees are eligible for overtime."
    Child B: "3.4.2 Rate: Overtime is compensated at 1.5x the regular hourly rate."
    Child C: "3.4.3 Threshold: Overtime applies to hours worked beyond 40 in a work week."

Query: "What is the overtime rate?"
  → Child B matches most closely (embedding similarity).
  → ParentDocumentRetriever returns the PARENT section (all of 3.4).
  → Model sees eligibility, rate, threshold, and approval together.
  → Complete, context-rich answer.
```

### Code

```python
from langchain.retrievers import ParentDocumentRetriever
from langchain.storage import InMemoryStore
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings
from langchain_community.document_loaders import PyPDFLoader

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# Load source documents
loader = PyPDFLoader("./documents/employee-handbook.pdf")
pages = loader.load()

# Two splitters: one for child chunks (small), one for parent chunks (large)
child_splitter = RecursiveCharacterTextSplitter(
    chunk_size=200,    # small children for precise retrieval
    chunk_overlap=20,
)
parent_splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,   # large parents for complete context
    chunk_overlap=100,
)

# Storage:
# - vectorstore: stores child chunk embeddings (searchable)
# - docstore: stores full parent chunks (returned to caller)
vectorstore = Chroma(
    collection_name="child_chunks",
    embedding_function=embeddings,
    persist_directory="./chroma_parent_retriever",
)
docstore = InMemoryStore()   # use Redis for production

# Build the retriever
parent_retriever = ParentDocumentRetriever(
    vectorstore=vectorstore,      # where child chunks are indexed
    docstore=docstore,            # where parent chunks are stored
    child_splitter=child_splitter,
    parent_splitter=parent_splitter,
)

# Index the documents (only needs to run once)
parent_retriever.add_documents(pages)
print("Indexed with parent-child structure")

# Retrieve: searches child chunks, returns parent chunks
results = parent_retriever.invoke("What is the overtime rate?")
for doc in results:
    print(f"Parent chunk length: {len(doc.page_content)} chars")
    print(doc.page_content[:300])
# The returned Document is the full parent section, not the small child chunk
```

### When to Use ParentDocumentRetriever

- When answers require understanding the section surrounding a specific sentence.
- When small chunks give precise matches but incomplete answers.
- When document structure is hierarchical (sections with subsections).

## Combining Retrievers

For maximum coverage, combine all three:

```python
from langchain.retrievers import ContextualCompressionRetriever, MultiQueryRetriever
from langchain.retrievers.document_compressors import LLMChainExtractor

# Layer: MultiQuery → ParentDocument → ContextualCompression
# Each layer improves a different dimension of quality

# Step 1: MultiQuery expands vocabulary coverage
multi_retriever = MultiQueryRetriever.from_llm(
    retriever=parent_retriever,     # inner retriever returns parent chunks
    llm=ChatOpenAI(model="gpt-4o-mini", temperature=0),
)

# Step 2: Compression filters irrelevant content from parent chunks
compressor = LLMChainExtractor.from_llm(
    ChatOpenAI(model="gpt-4o-mini", temperature=0)
)
full_pipeline_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=multi_retriever,
)

# Usage
final_docs = full_pipeline_retriever.invoke("How do I qualify for overtime?")
# 1. MultiQuery generates 3-4 rephrased versions of the question
# 2. Each rephrase searches the child chunk index
# 3. Matching child chunks are replaced by their parent sections
# 4. Compressor extracts only the directly relevant sentences
```

## Common Pitfalls

| Pitfall                                                  | What goes wrong                                   | Fix                                                                                    |
| -------------------------------------------------------- | ------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Using MultiQueryRetriever without logging                | You can't see what queries were generated         | Enable logging: `logging.getLogger("langchain.retrievers.multi_query").setLevel(INFO)` |
| High k with ContextualCompression                        | Many LLM calls for compression; expensive         | Use k=4-6 as base retrieval; compression handles the rest                              |
| ParentDocumentRetriever with InMemoryStore in production | Parent chunks lost on restart                     | Use Redis or a persistent docstore in production                                       |
| Stacking all three retrievers for every query            | Significant latency and cost increase             | Use the simplest retriever that achieves your quality target                           |
| Not testing on real user queries                         | Advanced retrievers tuned for wrong failure modes | Collect 50-100 real queries before choosing retrievers                                 |

---

## Mini Summary

- MultiQueryRetriever rephrases the query multiple ways to overcome vocabulary mismatch.
- ContextualCompressionRetriever strips irrelevant content from each retrieved chunk.
- ParentDocumentRetriever indexes small chunks for precision, returns large parents for context.
- These retrievers address specific root causes — match the retriever to the failure mode.
- Combining retrievers improves quality but increases cost; use the simplest solution that works.
